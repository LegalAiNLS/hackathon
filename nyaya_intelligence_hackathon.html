01 — Problem statement

India's justice system is flying blind

"No police station, judge, or policymaker currently has a unified, data-driven view of how cases move from FIR to conviction — or why they don't."

Karnataka registers 250,000+ FIRs annually. Investigation quality is invisible. Charge sheet filing deadlines under BNSS are routinely missed. Conviction rates vary wildly across stations. Policy reforms are made on anecdote, not evidence.

🔴

Data fragmentation

FIR data, chargesheet data, court records, and outcome data live in 4+ separate, incompatible systems. No single entity sees the full chain.

⚠️

Procedural non-compliance

BNSS mandates chargesheet filing within 60/90 days. No automated tracking exists. Officers learn of violations only at court hearings.

📊

Zero predictive capacity

Prosecutors file cases with no systematic analysis of how similar cases have fared. Investigation quality cannot be quantified or compared.

⚖️

No performance accountability

Police station performance on conviction rates, delay, and compliance is never measured. Promotion systems are disconnected from outcomes.

03 — Technical architecture

Production-grade system design

Click each layer to expand full implementation detail.

Data layer — ingestion, OCR, normalisation
▼
Python Scrapy Playwright (dynamic pages) Tesseract 5 OCR Apache Airflow DAGs PostgreSQL + JSONB AWS S3 (raw PDFs)

Sources: Karnataka Police CCTNS portal, eCourts API (v2), High Court of Karnataka judgment portal, Supreme Court NJDG. PDF/scanned FIRs processed via Tesseract with confidence scoring — low-confidence OCR zones flagged for human review rather than silently passed.

Noisy text strategy: Pre-processing pipeline: deskew → binarize → denoise → OCR → regex post-correction for common legal abbreviations (s/o, d/o, IPC § → BNS §). Section number normaliser maps legacy IPC sections to new BNS 2023 equivalents using a curated lookup table.

Schema: Canonical FIR JSON schema: fir_id, ps_code, district, date_registered, crime_type, accused[], victim[], bns_sections[], evidence_items[], chargesheet_date, court_case_id, outcome. Versioned schema migrations via Alembic.

Processing layer — NLP pipeline and knowledge graph
▼
InLegalBERT (IITM) spaCy 3.x NER Sentence-Transformers FAISS IndexFlatIP Neo4j Knowledge Graph Celery + Redis

NLP pipeline stages: (1) Language detection (Kannada/Hindi/English) → translation if non-English. (2) Section extraction: regex + NER hybrid to extract BNS section numbers. (3) Crime type classification: 23-class fine-grained taxonomy (IPC_to_BNS mapped). (4) Severity scoring: section severity index × accused count × weapon presence. (5) Summarisation: extractive summary for case card display.

Knowledge graph: Neo4j nodes: FIR → Chargesheet → Court_Case → Judgment. Edges: filed_under, investigated_by, adjudicated_at, results_in. Enables graph queries like "all acquittals where section 302 BNS was paired with section 34 at Bangalore South district in 2022-24."

Embeddings: Each FIR summary embedded using multilingual-e5-large into 1024-dim vector. Stored in FAISS. At query time, new FIR embedded and top-10 nearest neighbours retrieved by cosine similarity. Fallback: BM25 keyword search for very short / noisy FIRs.

ML layer — conviction prediction, explainability, bias detection
▼
XGBoost / LightGBM SHAP v0.44 scikit-learn Fairlearn (bias) MLflow tracking BentoML serving

Conviction prediction model: Binary classification (convicted / not convicted). Features: crime_type_encoded (one-hot, 23 classes), bns_section_severity, evidence_count_proxy (count of evidence_items in FIR), investigation_duration_days, chargesheet_filed_ontime (bool), accused_count, repeat_offender (bool), location_tier (1/2/3 city), district_baseline_conviction_rate. Trained on 50K Karnataka cases (2018-2023) from eCourts judgment data.

Model explainability (critical for legal adoption): Every prediction served with SHAP waterfall chart showing top-5 features pushing toward/away from conviction. Judges and prosecutors must be able to interrogate why the score is what it is. Model is explicitly positioned as a "decision support" tool, not a decision-making tool. Disclaimer displayed on every prediction screen.

Bias detection: Fairlearn demographic parity checks across religion, caste (where data exists), and district. Regular audits using Aequitas bias audit tool. Disparate impact reports generated quarterly for the transparency dashboard. Any model with demographic parity ratio < 0.8 is flagged for human review before deployment.

Backend — API, microservices, scheduler
▼
FastAPI (Python 3.11) PostgreSQL 15 Redis 7 (cache + queue) Celery beat (scheduler) Docker + Kubernetes Nginx + Traefik

Microservices: ingestion-service (scraper + OCR), nlp-service (NLP pipeline), prediction-service (ML inference), deadline-service (BNSS compliance engine), analytics-service (aggregation + reporting), auth-service (JWT, RBAC). Services communicate via gRPC internally, REST externally.

Deadline engine: Celery beat runs daily. Queries all open FIRs without chargesheet. Computes days_remaining = deadline_date − today. Sends alerts at 75% consumed (amber), 90% consumed (red), overdue (critical). Alerts via: dashboard notification, email to SHO, SMS via MSG91 API. All alerts logged to audit table.

Rate limiting + security: Sensitive prediction endpoints rate-limited to 100 req/min per API key. All PII fields (victim name, address) encrypted at rest (AES-256). Role-based access: POLICE role sees own station data only; DISTRICT_ADMIN sees district; DGP sees state; JUDICIARY sees court-assigned cases only.

Infrastructure — cloud, vector DB, data pipeline
▼
AWS GovCloud (ap-south-1) Pinecone (prod) / FAISS (dev) Apache Airflow on MWAA AWS RDS PostgreSQL AWS S3 (data lake) CloudWatch + PagerDuty

Data pipeline DAGs: daily_fir_scrape → ocr_process → nlp_classify → embed_and_index → model_inference → alert_check → dashboard_refresh. Full pipeline runs nightly at 02:00 IST. Incremental refresh runs every 6 hours for deadline alerts.

Data residency: All data stored in AWS ap-south-1 (Mumbai). Backup to ap-south-2 (Hyderabad). No data leaves India. DPDP Act compliance: data minimisation enforced — only fields necessary for analytics retained. Right-to-erasure hooks implemented for citizen portal.

Built around BNS, BNSS, and DPDP

BNSS §193: Supplementary investigation reports to magistrate. BNSS §176: Magistrate may order investigation by any police officer.

Chargesheet
BNSS §193(1): 60 days for non-bailable offences (magistrate court); 90 days for Sessions court cases. Bail by default on overrun.
60 / 90 days
↓
Cognizance & Trial
BNSS §258: Sessions court must frame charges within 60 days. BNSS §290: Evidence of witnesses on affidavit permitted.
60 days
↓
Judgment
BNSS §392: Judgment to be pronounced within 45 days of arguments. Total trial target: 3 years under BNSS fast-track provisions.
45 days

🔒 DPDP Act 2023 alignment

All victim/witness PII encrypted. Data minimisation enforced — no unnecessary personal data ingested. Consent management for citizen-facing portal. Breach notification workflow implemented.

⚖️ Ethical AI guardrails

Conviction score displayed as "decision support probability" with mandatory disclaimer. Model cannot be used as sole basis for arrest, charge, or bail decisions. SHAP explanations mandatory on all predictions.

🔭 Bias monitoring

Quarterly fairness audits. Disparate impact detection across demographics. Any bias finding triggers automatic model review. Audit reports published to transparency dashboard.

07 — Winning pitch structure

2-minute pitch script

Step 1 · Hook · 20 seconds

"Every 1.6 seconds, an FIR is filed somewhere in India. Only half will ever result in conviction. Nobody — not the police, not the judge, not the policymaker — has a clear, data-driven picture of why."

Open with the scale. Make the number personal — it's not a statistic, it's 10 million victims a year whose cases disappear into the system.

Step 2 · Insight · 20 seconds

"The data exists. CCTNS, eCourts, and High Court portals hold millions of records — but they are siloed, unstructured, and unanalysed. The system has the raw material for intelligence. It just lacks the engine."

This is the key insight: this is not a data collection problem. It is a data activation problem. Nyaya Intelligence is that engine.

Step 3 · Demo · 40 seconds

"Watch: I upload a raw FIR PDF. In under three seconds, Nyaya extracts the BNS sections, computes a conviction probability of 68% with a plain-English explanation of why — and shows me five similar cases from Karnataka courts."

Run the live demo here. The technology does the talking. Let the panel feel the speed and clarity.

Step 4 · Impact · 20 seconds

"A 5% improvement in chargesheet compliance across Karnataka is 12,500 fewer bail-by-default releases per year. That's 12,500 cases where justice doesn't walk out the door on a technicality."

Make the impact concrete and quantifiable. Panels respond to specific numbers tied to real legal consequences.

Step 5 · Scale · 20 seconds

"Karnataka is the proof of concept. The architecture works for any Indian state in 90 days. We partner with National Crime Records Bureau for national rollout. This is not a dashboard — it is the nervous system of India's criminal justice infrastructure."

End with ambition. The judges are evaluating for national impact, not just local solutions.
